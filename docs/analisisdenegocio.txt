Esta primera versión tiene como objetivo comprender completamente el funcionamiento del negocio antes de escribir cualquier línea de código.

Durante esta etapa no se desarrollará ninguna funcionalidad del sistema. En cambio, se documentará el dominio del problema, los procesos operativos, los actores involucrados, las reglas de negocio y la arquitectura conceptual del software.

El objetivo principal es garantizar que todas las decisiones técnicas posteriores se basen en un entendimiento profundo del proceso empresarial y no únicamente en aspectos tecnológicos.

Objetivos

Al finalizar la versión 0.1 deberíamos ser capaces de responder preguntas como:

¿Qué problema resuelve el sistema?
¿Quiénes utilizan el sistema?
¿Qué hace cada usuario?
¿Qué información maneja?
¿Qué reglas de negocio existen?
¿Qué sucede cuando ocurre un caso excepcional?
¿Cómo fluye la información desde que llega una fotografía hasta que se genera una liquidación?
¿Qué entidades existirán dentro de la base de datos?
¿Qué módulos tendrá la aplicación?
¿Cómo será la arquitectura general?

Si alguna de estas preguntas aún no tiene respuesta, significa que todavía no estamos listos para programar.

Documento 5
Reglas del negocio

Este documento será gigantesco.

Y probablemente el más importante del proyecto.

Cada regla tendrá el siguiente formato.

RN-001

Nombre

Descripción

Actores involucrados

Entradas

Proceso

Resultado

Observaciones

Ejemplo.

RN-001

Los contadores nunca se reinician al finalizar un mes.

RN-002

La ganancia del período corresponde únicamente a la diferencia entre los contadores del período anterior y el actual.

RN-003

Cuando existe un reinicio de contadores se debe registrar un movimiento especial.

RN-004

Los días fuera de operación afectan el cálculo del fee.

RN-005

Los ajustes aprobados por el delegado nunca eliminan el dato original.

RN-006

Toda liquidación cerrada queda bloqueada para edición.

Y así sucesivamente.

No me sorprendería terminar con más de 100 reglas.

🎯 Objetivos de calidad

Desde la primera versión el proyecto deberá seguir principios de desarrollo profesional:

Arquitectura por capas.
Código limpio (Clean Code).
Principios SOLID.
Diseño orientado al dominio (DDD ligero).
Control de versiones con Git.
Documentación técnica.
Auditoría de cambios.
Escalabilidad.
Parametrización de reglas de negocio.
Seguridad basada en roles.
Modularidad.
¿Qué esperamos lograr con la v0.1?

Al finalizar esta fase no existirá una aplicación funcional, pero sí existirá algo mucho más importante: un conocimiento estructurado del negocio.

Esta documentación será la base sobre la cual se diseñará la base de datos, la API, el frontend y cada una de las funcionalidades futuras. Cualquier cambio importante en el sistema deberá comenzar actualizando estos documentos antes de modificar el código.

El objetivo no es simplemente desarrollar una aplicación web, sino construir una plataforma empresarial capaz de digitalizar y optimizar un proceso operativo real, manteniendo la trazabilidad, la integridad de la información y la flexibilidad necesaria para adaptarse a cambios futuros en las reglas del negocio.
